# プロジェクト概要：Robloxアセット一括アップローダー（Tauriデスクトップアプリ）

## 背景・目的

Robloxのゲーム開発者は、Roblox Open Cloud Assets APIを通じてアセット（テクスチャ・サウンド・モデルなど）をアップロードできる。しかし既存のツールはCLI操作が必要であり、コマンドラインに慣れていない一般クリエイターには敷居が高い。

このプロジェクトでは、**CLIなしでGUIから操作できるクロスプラットフォームのデスクトップアプリ**を作る。WindowsとmacOSの両方で動作すること。

---

## 技術スタック

- **フレームワーク**：Tauri（Rust + HTML/CSS/JavaScript）
- **バックエンド**：Rust — `reqwest` で Roblox Open Cloud Assets REST API を直接呼び出す
- **フロントエンド**：バニラ HTML / CSS / JavaScript（フレームワーク不要）
- **配布形式**：`.exe`（Windows）/ `.dmg`（macOS）

### 主要な依存クレート

```toml
[dependencies]
reqwest = { version = "0.12", features = ["multipart", "json", "stream"] }
serde = { version = "1.0", features = ["derive"] }
serde_json = "1.0"
tokio = { version = "1", features = ["full"] }
toml = "0.8"        # ロックファイル読み書き用
keyring = { version = "3", features = ["apple-native", "windows-native", "sync-secret-service", "crypto-rust"] }
# apple-native: macOS Keychain
# windows-native: Windows Credential Manager
# sync-secret-service + crypto-rust: Linux Secret Service (GNOME Keyring / KWallet)
```

### 使用するRoblox Open Cloud エンドポイント

| 操作 | メソッド | エンドポイント |
|---|---|---|
| アセット作成 | `POST` | `https://apis.roblox.com/assets/v1/assets` |
| アセット更新 | `PATCH` | `https://apis.roblox.com/assets/v1/assets/{assetId}` |
| オペレーション取得 | `GET` | `https://apis.roblox.com/assets/v1/operations/{operationId}` |

- 認証は `x-api-key` ヘッダーで行う
- リクエストは `multipart/form-data` 形式で、`request`（JSON文字列）と `fileContent`（バイナリ）の2パートで構成
- 1ファイルあたり最大20MB
- アセットタイプごとにアップロード月間制限あり（ID認証済み: 100件/月、未認証: 10件/月）※オーディオの場合

## MVP機能

### 1. フォルダ・ファイル選択による一括アップロード
- フォルダ単位でもファイル単位でもアップロード可能
- フォルダ・ファイルともにドラッグ＆ドロップまたはダイアログから選択できる
- アップロードは以下の2ステップで行う：
  1. **プレビュー**: ドロップまたは選択後、対象ファイルの一覧をサムネイル・ファイル名・サイズ・アセットタイプ付きで表示する。ロックファイルに同名ファイルが既に存在する場合は「上書き」マークを表示する。この段階でユーザーは個別にファイルを除外できる
  2. **アップロード実行**: ユーザーが「アップロード」ボタンを押して初めてアップロードを開始する。ボタンを押すまではサーバーへのリクエストは一切行わない
- ファイルごとに結果を表示：成功 / 失敗 / 割り当てられたアセットID
- `reqwest` の `Body::wrap_stream` を活用し、ファイルごとのアップロード進捗をUIに反映する

### 2. ロックファイルによるアセットID管理
- ファイル名 ↔ アセットIDの対応を `assets.lock.toml` に保存
- アセットの同一性は**ファイル名（拡張子含む）**で判断する（パスは無視）
- 異なるフォルダに同名ファイルがある場合（例：`textures/wood.png` と `materials/wood.png`）は競合としてユーザーに警告し、どちらを使うか選択させる
- 同じファイル名がすでにロックファイルに登録されている場合は、上書きするかキャンセルするかをユーザーに確認する
- 上書き時は既存のアセットIDに対して **Update API（PATCH）** を使用し、アセットIDを変えずに内容だけを更新する（新規作成してID差し替えは行わない）
- これにより、Luauコード側の `rbxassetid://` 参照が変わらず、コーダーは何も修正する必要がない
- ロックファイルはプロジェクトフォルダに生成・保存される

### 3. Luauコードの自動生成
- ロックファイルからLuauモジュールを生成し、コーダーが名前でアセットを参照できるようにする
- アセットタイプ別（Image / Audio / Model）にテーブルをネストする。分類は拡張子から自動判定する
- コーダーはモジュールを `require` するだけでよく、IDを手動で管理する必要がない（例：`Assets.Image.wood_floor`）
- ファイルを差し替えて再アップロードすると、ロックファイルと生成モジュールが自動更新される。コーダー側のコードは変更不要
- アップロード完了後に自動的に `Assets.lua` を再生成する

### 4. APIキー管理・クリエイター管理
- Roblox Open Cloud APIキーを `keyring` クレートでOSネイティブのセキュアストレージに保存
  - **macOS**: Keychain（`apple-native` feature）
  - **Windows**: Credential Manager（`windows-native` feature）
  - **Linux**: Secret Service API — GNOME Keyring / KWallet（`sync-secret-service` + `crypto-rust` feature）
- APIキーはプレーンテキストやファイルには一切保存しない
- `keyring::Entry::new("rbx-asset-uploader", "api-key")` のようにサービス名でエントリを識別する
- **注意**: Tauri公式の Stronghold プラグインはv3で非推奨・削除予定のため採用しない
- **クリエイタープロファイル**として複数のGroupID（およびUserID）を登録・命名して保存できる（プロファイル情報は機密性が低いため `tauri-plugin-store` または設定ファイルに保存）
- アップロード前にプロファイルをドロップダウンで切り替えられる
- メイン画面のボタンからいつでも設定画面を呼び出せる
- クリエイタープロファイルが一つも登録されていない場合（初回起動時など）は自動的に設定画面を表示する

### 5. エラーハンドリングとリトライ
- HTTPステータスコードに基づくエラー分類（認証エラー / レート制限 / サーバーエラー / バリデーションエラー）
- レート制限（429）を受けた場合は `Retry-After` ヘッダーを尊重して自動リトライ
- ネットワークエラーやサーバーエラー（5xx）は指数バックオフ（1秒 → 2秒 → 4秒）で最大3回リトライ
- Operation取得（GetOperation）のポーリング：完了まで2秒間隔で最大30回（60秒）リトライ
- すべてのエラーをUI上でファイル単位で表示し、失敗したファイルのみ再アップロード可能

## ロックファイルの形式（`assets.lock.toml`）

```toml
[assets]

[assets.Image]
"wood_floor.png" = 12345678
"brick_wall.jpeg" = 22334455

[assets.Audio]
"bgm.mp3" = 87654321

[assets.Model]
"shelf.fbx" = 11223344
```

---

## 生成されるLuauモジュールの例（`Assets.lua`）

```lua
-- Auto-generated by Roblox Asset Uploader
-- Do not edit manually

local Assets = {
    Image = {
        wood_floor = "rbxassetid://12345678",
        brick_wall = "rbxassetid://22334455",
    },
    Audio = {
        bgm = "rbxassetid://87654321",
    },
    Model = {
        shelf = "rbxassetid://11223344",
    },
}

return Assets
```

## UI画面構成

1. **設定画面**：
   - APIキーの入力と保存（OSネイティブのセキュアストレージへ保存。ファイルには保存しない）
   - クリエイタープロファイルの管理（追加・編集・削除）
   - プロファイルごとに名前・タイプ（user / group）・IDを設定
2. **メイン画面**：
   - クリエイタープロファイルの切り替えドロップダウン（GroupID / UserIDを常時表示）
   - プロジェクトフォルダのセレクター
   - ドラッグ＆ドロップエリア（ファイル・フォルダ受付）
   - **プレビューリスト**（ドロップ/選択後に表示）：
     - ファイル名・サイズ・アセットタイプ
     - 画像（.png / .jpeg）：サムネイル表示
     - オーディオ（.mp3 / .ogg / .flac / .wav）：再生ボタン
     - 3Dモデル（.fbx）：アイコン表示のみ
     - ロックファイルに既存の場合は「上書き」バッジ表示
     - ファイルごとの除外チェックボックス
   - **「アップロード」ボタン**（プレビュー表示中のみ有効）
   - ファイルごとのアップロード進捗バー（アップロード実行後に表示）
   - アップロード結果（成功 / 失敗 / アセットID）
   - 失敗ファイルの再アップロードボタン

## サポートするアセットタイプ

| カテゴリ | 拡張子 | API上の assetType | 備考 |
|---|---|---|---|
| 画像 | `.png`, `.jpeg` | `Decal` | API上の名称は「Decal」だが、生成されるアセットIDはテクスチャ・パーティクル・UI画像など用途を問わず使用可能 |
| オーディオ | `.mp3`, `.ogg`, `.flac`, `.wav` | `Audio` | |
| 3Dモデル | `.fbx` | `Model` | |

> **注意**: `.bmp` と `.tga` はOpen Cloud APIで正常に動作しない報告があるため、MVPではサポート対象外とする。

## 開発ステップ

1. Tauriプロジェクトの初期化
2. `reqwest` + `tokio` を追加してビルドが通るか確認
3. APIキーの保存・読み込み（`keyring` クレートによるOS Keychain / Credential Manager / Secret Service連携）
4. Open Cloud REST APIを直接呼び出すアップロードモジュールの実装（Create / Update / GetOperation）
5. エラーハンドリング・リトライロジックの実装
   - HTTPレスポンスのステータスコードによるエラー分類：
     - `400` バリデーションエラー（不正なファイル形式・サイズ超過等）→ リトライ不可、ユーザーに原因を表示
     - `401` / `403` 認証エラー（APIキー無効・権限不足）→ リトライ不可、設定画面への誘導
     - `429` レート制限 → `Retry-After` ヘッダーの秒数だけ待ってから自動リトライ
     - `500` / `502` / `503` サーバーエラー → 指数バックオフ（1秒 → 2秒 → 4秒）で最大3回リトライ
     - ネットワーク接続エラー（タイムアウト・DNS解決失敗等）→ サーバーエラーと同じリトライ戦略
   - リトライはファイル単位で独立して行い、成功済みファイルには影響しない
   - すべてのリトライ失敗後、該当ファイルを「失敗」状態としてUIに表示し、手動で再アップロード可能にする
   - Operation取得（GetOperation）のポーリング：完了まで2秒間隔で最大30回（60秒）リトライ
6. ロックファイルの読み書き（`toml` クレート）
7. フロントエンドUIの実装
8. Luauコード生成
9. ビルドと配布設定（Windows / macOS）

**ステップ1：Tauriプロジェクトの初期化**から始めてください。
